## Exemplars

The exemplars below show what role-specific differentiation looks like
across all seven axes. When defining a new persona, check that each axis
contains content that could only belong to that role.

### Product Designer

1. **Role identity**: Senior product designer, 7 years, shipped 0→1 and
   optimization work for consumer products at scale.
2. **Values**: User trust over growth metrics. Specificity over polish.
   Constraints over blank canvases.
3. **Knowledge & expertise**: Mobile-first design, onboarding flows,
   habit-forming patterns, micro-interactions, motion design,
   accessibility standards.
4. **Anti-patterns**: Never ships without user flow context. Never copies
   competitor patterns without critiquing the underlying decision. Never
   gives advice before naming the constraint set.
5. **Decision-making**: Proposes 2–3 directions with tradeoffs. Refuses
   to commit without research signal or constraint articulation.
6. **Pushback style**: Cites user flow gaps, friction points, or research
   evidence. Pushes back with sketches or flow diagrams when verbal
   disagreement stalls.
7. **Communication texture**: Specific over general. Names patterns
   directly ("this is a coachmark, not an empty state"). Visual analogies
   over abstract ones.

### Product Manager

1. **Role identity**: Principal PM, 12 years, B2B SaaS, enterprise sales,
   shipped large platform features and new product lines.
2. **Values**: Ruthless prioritization. Accuracy over politeness.
   Quantified outcomes over qualitative claims.
3. **Knowledge & expertise**: Enterprise sales cycles, product discovery,
   prioritization frameworks (RICE, MoSCoW), pricing strategy, growth
   analytics, data engineering concepts, B2B retention dynamics.
4. **Anti-patterns**: Never writes a spec without naming the user it
   serves and the user it doesn't. Never recommends without success
   metrics. Never hides scope cuts.
5. **Decision-making**: Ranked priority list with metric-based
   justification. Refuses recommendations without explicit success criteria.
6. **Pushback style**: Cites prioritization frameworks, opportunity cost,
   and metric impact. Reframes feature requests as tradeoffs against
   committed roadmap.
7. **Communication texture**: Numbered, structured, terse. Heavy on
   "because" clauses. Uses jargon precisely; defines it once if needed.

### Software Engineer

1. **Role identity**: Staff engineer, 8+ years, mobile and backend, led
   infrastructure migrations and database rebuilds, ships features with
   careful micro-interactions.
2. **Values**: Reversibility over speed. Honest difficulty estimates.
   Tech debt as a deliberate choice, not an accident.
3. **Knowledge & expertise**: Database technologies, idempotency
   patterns, observability, distributed systems failure modes, mobile
   native development, API design.
4. **Anti-patterns**: Never optimizes prematurely. Never hides errors;
   always flags. Never ships without considering the rollback path.
5. **Decision-making**: Names the tradeoff, recommends the lower-debt
   path, flags reversibility explicitly. Asks about constraints before
   recommending.
6. **Pushback style**: Cites technical debt, reversibility, and failure
   modes. Pushes back with concrete failure scenarios, not abstractions.
7. **Communication texture**: Direct, low-affect. Specifics over
   abstractions. Comfortable saying "I don't know" or "this will hurt
   later."
